# 시스템 기획서

## 프로젝트명
`Air DJ: 손짓으로 믹스하는 나만의 무대`

## 콘텐츠 개요
`Air DJ`는 Windows PC 웹캠과 `MediaPipe Hands` 기반 손 추적을 사용하는 인터랙티브 콘텐츠다. 사용자는 복잡한 장비 없이 화면 앞에서 손을 움직이는 것만으로 음악의 리듬, 분위기, 효과를 실시간으로 바꾸며 DJ 퍼포먼스를 체험한다. 초보자도 즉시 반응을 이해할 수 있는 직관적인 제스처와 시각적 피드백을 중심으로, 짧은 체험 시간 안에 성취감과 몰입감을 전달하는 것을 목표로 한다.

## 핵심 컨셉
### 한 줄 정의
- `MediaPipe Hands`로 인식한 손 제스처를 DJ 믹싱 조작으로 변환해, 초보자도 즉시 음악 퍼포먼스를 만드는 핸드 트래킹 인터랙티브 콘텐츠

### 지향 경험
- 복잡한 장비 없이 카메라 앞에 서는 것만으로 바로 시작할 수 있어야 한다.
- 사용자는 손동작이 음악과 화면에 즉시 반응하는 쾌감을 느껴야 한다.
- 짧은 체험 시간 안에 "내가 직접 무대를 만들었다"는 성취감을 전달해야 한다.

### 사용자 가치
- 음악 지식이 없어도 쉽게 참여할 수 있다.
- 어떤 제스처를 해도 결과가 자연스럽고 그럴듯하게 느껴져야 한다.
- 시각 효과와 사운드 변화가 함께 반응해 퍼포먼스하는 재미를 강화해야 한다.

### 차별 포인트
- 손을 컨트롤러처럼 사용하는 비접촉 방식의 DJ 체험
- 음악 믹싱을 단순 제스처 기반 인터랙션으로 재해석한 접근성 높은 구조
- 전시, 이벤트, 체험존에 적용하기 쉬운 짧고 강한 몰입형 퍼포먼스 포맷

## 핵심 메커니즘
### 1. 음악 선택
#### 목적
- 사용자가 체험 시작 전 재생할 음악 또는 루프 세트를 직관적으로 선택한다.

#### 프로토타입 방식
- 시작 화면에서 2~3개의 음악 카드 중 하나를 선택한다.
- 선택 방식은 마우스 클릭 또는 운영자 보조 입력을 기본으로 한다.
- 선택이 완료되면 곧바로 플레이 화면으로 전환한다.

#### 구현 체크 포인트
- 음악 카드별 제목 또는 구분 텍스트가 보인다.
- 선택된 음악이 시각적으로 강조된다.
- 선택 후 지정된 트랙 또는 루프 세트가 로드된다.

### 2. 음악 재생 시작
#### 목적
- 사용자가 손을 인식시키는 순간 체험이 바로 시작되도록 한다.

#### 프로토타입 방식
- 플레이 화면 진입 후 대기 상태를 유지한다.
- `MediaPipe Hands`에서 한 손 또는 양손이 안정적으로 인식되면 기본 비트와 배경 루프를 재생한다.
- 손 인식 전에는 안내 UI를 유지한다.

#### 구현 체크 포인트
- 손 미인식 상태에서 시작 안내 문구가 노출된다.
- `MediaPipe Hands` 인식 성공 시 음악 재생이 시작된다.
- 재생 시작과 동시에 UI가 활성 상태로 전환된다.

### 3. 제스처 기반 디제잉
#### 목적
- 손동작을 단순한 DJ 조작으로 변환해 초보자도 즉시 결과를 체험하게 한다.

#### 프로토타입 대상 제스처
- 오른손 위/아래 이동: 필터 열기/닫기
- 왼손 위/아래 이동: 볼륨 또는 레이어 양 조절
- 손 좌우 이동: 크로스페이더 이동
- 양손 벌리기/모으기: 공간감 또는 빌드업 강도 조절
- 손목 회전: 스크래치 또는 FX 강도 변화
- 주먹 쥐기/펴기: 원샷 효과음 트리거

#### 구현 체크 포인트
- 각 제스처마다 1개의 명확한 사운드 결과가 연결된다.
- 제스처 입력 시 음악 변화가 즉시 들린다.
- 잘못된 입력이 들어와도 사운드가 과하게 깨지지 않는다.

### 4. UI 시각 피드백
#### 목적
- 사용자가 지금 어떤 제스처를 인식시켰고 어떤 효과가 적용됐는지 바로 이해하게 한다.

#### 프로토타입 방식
- 화면 하단 또는 좌우에 제스처 가이드를 고정 노출한다.
- 현재 인식된 손 위치와 활성 제스처를 하이라이트한다.
- 필터, 볼륨, 크로스페이더, FX 상태를 단순 게이지 또는 다이얼로 보여준다.
- 비트에 맞춰 파형, 글로우, 파티클이 반응한다.

#### 구현 체크 포인트
- 현재 인식된 손 또는 제스처가 화면에서 확인된다.
- 사운드 파라미터 변화가 UI 수치 또는 게이지로 보인다.
- 음악 반응과 시각 효과 반응이 동시에 느껴진다.

### 5. 체험 종료 또는 반복
#### 목적
- 짧은 체험 세션을 명확하게 마무리하고 다음 사용자 흐름으로 이어지게 한다.

#### 프로토타입 방식
- 일정 시간 경과 또는 운영자 입력으로 체험을 종료한다.
- 종료 시 음악을 페이드아웃하고 결과 연출 또는 재시작 안내를 노출한다.
- 이후 시작 화면 또는 음악 선택 화면으로 복귀한다.

#### 구현 체크 포인트
- 종료 조건이 명확하게 동작한다.
- 종료 시 사운드와 UI가 자연스럽게 초기화된다.
- 다음 사용자가 바로 다시 시작할 수 있는 상태로 복귀한다.

## 기술 명세
프로토타입 버전에서는 시스템별 핵심 진입 클래스를 1개씩만 두고, 세부 데이터/설정/뷰 책임은 해당 클래스 내부 직렬화 필드, private 헬퍼 구조, Unity 컴포넌트 참조로 흡수해 전체 공개 클래스 수를 6개로 유지한다.

### 1. 경험 흐름 및 씬 제어 시스템
- 생성할 클래스 구조
  - `UE_ExperienceFlowController` : `MonoBehaviour`, 부트스트랩, 타이틀, 음악 선택, 플레이 시작, 종료, 재시작, 세션 상태 초기화를 통합 제어 담당
- 검증원칙
  - 음악 선택부터 플레이, 종료, 재시작까지의 흐름이 한 번의 세션에서 끊기지 않아야 한다.
  - 종료 후 재진입 시 이전 세션 데이터가 남지 않아야 한다.
- 응답원칙
  - 상태 전환은 즉시 반영하되 시각적 전환과 오디오 페이드는 자연스럽게 이어져야 한다.
  - 예외 발생 시 마지막 정상 상태를 유지하고 운영자 로그를 남긴다.

### 2. 음악 선택 및 오디오 재생 시스템
- 생성할 클래스 구조
  - `UE_MusicSelectionController` : `MonoBehaviour`, 음악 카드 표시, 선택 결과 반영, 트랙 메타데이터 보관, 선택 트랙 로딩 및 루프 재생 요청을 통합 담당
- 검증원칙
  - 한 번에 하나의 음악만 선택되어야 하며 선택 결과가 플레이 시작까지 유지되어야 한다.
  - 필수 오디오 클립이 누락된 경우 안전한 기본 트랙으로 대체되어야 한다.
- 응답원칙
  - 선택 직후 강조 표시와 로딩 상태가 바로 반영되어야 한다.
  - 프로토타입에서는 선로딩 우선으로 시작 지연을 최소화한다.

### 3. 웹캠 입력 및 손 추적 시스템
- 생성할 클래스 구조
  - `UE_WebcamTrackingController` : `MonoBehaviour`, 웹캠 시작, 프레임 공급, `MediaPipe Hands` 브리지 연결, 양손 랜드마크 변환, 좌표 보정, 신뢰도 관리까지 통합 담당
- 검증원칙
  - 웹캠 장치가 없거나 `MediaPipe Hands` 초기화에 실패해도 앱이 멈추지 않아야 한다.
  - `MediaPipe Hands`의 양손 랜드마크 결과가 좌우 손 구분과 함께 안정적으로 변환되어야 한다.
  - 양손 데이터는 독립적으로 갱신되며 좌우 반전과 노이즈 보정이 일관되어야 한다.
- 응답원칙
  - 플레이 시스템에는 `MediaPipe Hands` 원본값이 아닌 정규화된 안정 좌표만 전달한다.
  - 신뢰도 하락 시 즉시 튀는 값 대신 마지막 안정값을 짧게 유지한 뒤 미인식 상태로 전환한다.
  - 외부 추적 브리지 오류 시 플레이 흐름 시스템에 미인식 상태를 명시적으로 전달한다.

### 4. 제스처 판별 및 디제잉 제어 시스템
- 생성할 클래스 구조
  - `UE_GestureDjController` : `MonoBehaviour`, 손 데이터 해석, 제스처 판별, 임계값 관리, DJ 파라미터 매핑, 추적 손실 복귀 처리를 통합 담당
- 검증원칙
  - 같은 입력에 대해 같은 제스처 결과가 반복 가능하게 나와야 한다.
  - 잘못된 입력이나 겹치는 제스처가 들어와도 사운드가 과하게 깨지지 않아야 한다.
- 응답원칙
  - 연속형 제스처와 이벤트형 제스처를 분리 처리한다.
  - 모든 파라미터는 안전 범위 안에서만 반영하고, 추적 손실 시 기본값 쪽으로 완만하게 복귀한다.

### 5. 플레이 HUD 및 반응형 비주얼 시스템
- 생성할 클래스 구조
  - `UE_PlayFeedbackController` : `MonoBehaviour`, HUD, 상태 텍스트, 제스처 가이드, 게이지, 반응형 파형/글로우/입자 연출 갱신을 통합 담당
- 검증원칙
  - 현재 인식 상태와 주요 오디오 파라미터가 화면에서 직관적으로 확인되어야 한다.
  - 시각 효과는 오디오 반응보다 눈에 띄게 늦지 않아야 하며 프레임 저하를 유발하지 않아야 한다.
- 응답원칙
  - 플레이어 학습에 필요한 핵심 정보만 우선 노출한다.
  - 성능 저하 시 입자량, 후처리, 보조 연출부터 단계적으로 축소한다.

### 6. 운영자 보조 및 공통 설정 시스템
- 생성할 클래스 구조
  - `UE_OperatorControlController` : `MonoBehaviour`, Old Input 기반 재시작, 강제 종료, 디버그 입력, 전역 튜닝값 참조, 공통 시스템 참조 연결을 통합 담당
- 검증원칙
  - 운영자 입력은 플레이어 제스처 입력과 충돌하지 않아야 한다.
  - 공통 설정은 씬 전환 이후에도 일관되게 참조되어야 한다.
- 응답원칙
  - 조정이 잦은 값은 인스펙터에서 즉시 튜닝 가능한 설정 데이터로 분리한다.
  - 운영자 명령은 플레이 흐름 시스템을 통해 일관된 경로로 처리한다.
